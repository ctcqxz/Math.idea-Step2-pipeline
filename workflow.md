### 步骤二 核心工作流程

1. 从 arXiv 上下载论文的 latex 代码（设计爬虫，批量下载）。
2. 对 latex 代码进行阅读和解析，输出其 ① 研究领域 ② 研究问题 ③ 主要结果 ④ 证明路线（简述，不超过 300 字）。
3. 提取论文中的全局假设（例如环是否有单位元、是否可交换）。
4. 切分并提取论文中所有含有核心步骤的定理（核心步骤是指具有本质困难的步骤，即og id的原料），以 json 数组格式输出。
5. 对 4. 中的每个定理都执行：提取其中每一个关键困难步骤。
6. 重要性审查：对 5. 中的每个步骤都进行重要性评分，0~100 分取值。若不足 95 分则告知原因及修改方案并返回 5. 。若循环次数 >10 则报错，作废。接下来的操作对每个步骤分别执行。
7. 询问这段证明中的核心技巧是什么，补全它对应的 idea 。
8. 一步一步分析这个证明人类是如何想到的，每次找寻一个核心 idea，更新一次当前的证明状态。
9. 综合比较 7. 和 8. 中的技巧，选取更优的那一个。
10. 将其转化为 json struct 格式，包含 ① 全局假设 ② 数学对象 ③ 条件假设 ④ 已知结论 ⑤ 证明目标。此即为 og id。全局假设仅选取需要的部分。
11. 简洁性核查：对每个 og id 进行简洁性评分，0~100 分取值。若不足 95 分则告知原因及修改方案并返回 7. 。若循环次数 >10 则报错，作废。
12. 准确性核查：对每个 og id 进行准确性评分，0~100 分取值。若不足 99 分则告知原因及修改方案并返回 7. 。若循环次数 >10 则报错，作废。
13. 输出 json struct 格式的 og id，即为最终成果。统计数量，运行时间。

### 注：

1. 重要性、简洁性、准确性核查中的循环次数以及及格线均可根据实际情况更改
2. 全局假设不一定需要，根据实际情况可舍去
3. 提示词要区分 system prompt 和 user prompt。
